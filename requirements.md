# 要件定義書: 逆算型・段階起床アラームアプリ

**Version:** 0.1  
**Status:** Draft  
**対象:** MVP要件定義  
**形式:** Markdown

---

## 1. 概要

本アプリは、ユーザーが指定した「起きたい時刻」から逆算し、その前の一定時間に複数のアラームを自動生成する目覚ましアプリである。

通常のアラームアプリのように「1回鳴って、止めたら終了」ではなく、ユーザーが一度目を覚ましても二度寝する可能性を前提に、起床目標時刻まで段階的に複数回アラームを鳴らす。

---

## 2. プロダクトコンセプト

### 2.1 コアコンセプト

> 起きたい時刻を決めるだけで、その前から段階的に何度も起こしてくれるアラーム。

### 2.2 設計思想

本アプリは、ユーザーの「起きたつもり」を信用しない。

そのため、アラーム鳴動中に **「起きた」ボタン** は表示しない。  
現在鳴っているアラームを止めても、同じ起床プラン内の未来のアラームはキャンセルされない。

例:

```text
起床目標: 07:00
設定: 60分前から5分間隔

06:00 鳴る → 今のアラームを止める
06:05 鳴る → 今のアラームを止める
06:10 鳴る → 今のアラームを止める
...
07:00 鳴る
```

---

## 3. 想定ユーザー

主な想定ユーザーは以下。

```text
- 目覚ましを1回止めても二度寝してしまう人
- 既存のスヌーズでは不安な人
- 毎朝複数のアラームを手動で設定している人
- 起床時刻から逆算してアラームを作るのが面倒な人
- 早朝予定、出社、登校、旅行、試験などで確実に起きたい人
```

---

## 4. MVPの目的

MVPでは、以下の仮説を検証する。

> 起きたい時刻だけを指定し、そこから逆算して複数アラームが自動生成される体験は、既存アラームより便利で継続利用される。

MVPでは、カレンダー連携や予定ベースの自動提案は含めない。  
まずは **「手動で起床時刻を指定するだけで、段階起床プランが作れる」** 体験に集中する。

---

## 5. スコープ

### 5.1 MVPに含めるもの

| 機能 | MVP判定 | 補足 |
|---|---:|---|
| 起きたい時刻の設定 | 含める | アプリの中核 |
| 起床時刻から逆算した複数アラーム生成 | 含める | 最重要機能 |
| デフォルトで「60分前から5分間隔」 | 含める | 初期体験をシンプルにする |
| 起床ウィンドウの調整 | 含める | 例: 30分、45分、60分、90分 |
| アラーム間隔の調整 | 含める | 例: 3分、5分、10分、15分 |
| デフォルト値の変更 | 含める | 継続利用に必要 |
| 一回限りの起床プラン | 含める | 単発予定向け |
| 曜日繰り返し | 含める | 平日・曜日指定など |
| 次回だけスキップ | 含める | 明日だけ不要、に対応 |
| 過ぎた一回限りアラームの整理 | 含める | 一覧を汚さない |
| 現在鳴っているアラームの停止 | 含める | ただし未来のアラームは止めない |
| 通知・アラーム権限チェック | 含める | 鳴らないリスクを下げる |
| テストアラーム | 含める | ユーザーの不安を減らす |

### 5.2 将来的に含めるもの

| 機能 | 判定 | 補足 |
|---|---:|---|
| カレンダー予定の表示 | 将来目標 | MVP後 |
| カレンダー予定から起床プラン作成 | 将来目標 | 予定開始のn時間前に起きる |
| アラーム未設定予定の通知 | 将来目標 | 誤通知対策が必要 |
| 繰り返しアラームの特定日無効化 | 将来目標 | カレンダー例外的な機能 |
| 祝日・休日スキップ | 将来目標 | 地域・カレンダー設定が必要 |
| 移動時間を考慮した起床提案 | 将来目標 | 予定場所・交通手段が必要 |
| 起床成功ログ | 将来目標 | 何回目で止めたか等 |
| 鳴動パターンの段階変化 | 将来目標 | 序盤は弱く、終盤は強く |
| ミッション解除 | 将来目標 | 計算、QR、歩数など |
| ウィジェット・ロック画面表示 | 将来目標 | OS実装確認後 |

### 5.3 MVPでは含めないもの

| 機能 | 判定 | 理由 |
|---|---:|---|
| 起きたボタン | omit | 二度寝リスクがあり、コンセプトと矛盾する |
| アラーム鳴動画面での残り一括停止 | omit | 眠い状態で押すとプロダクト価値を壊す |
| カレンダーの自動監視 | omit | 初期にはノイズが多い |
| 予定に対する自動アラーム生成 | omit | ユーザー意図の誤判定が起きやすい |
| 本格的なRRULE対応 | omit | 繰り返し仕様が複雑化しすぎる |
| AIによる予定解析 | omit | 初期には不要 |
| 睡眠トラッキング | omit | プロダクトの焦点がぼやける |
| スヌーズ | omit | 本アプリ自体が事前設計された段階アラームとして機能するため |

## 6. 用語定義

### 6.1 起床プラン / Wake Plan

ユーザーが設定する、1つの起床目標単位。

例:

```text
平日 07:00 に起きる
60分前から5分間隔で鳴らす
```

### 6.2 アラーム発生予定 / Alarm Occurrence

Wake Planから生成される個々のアラーム。

例:

```text
06:00
06:05
06:10
...
07:00
```

### 6.3 起床目標時刻

ユーザーが最終的に起きたい時刻。

例:

```text
07:00
```

### 6.4 起床ウィンドウ

起床目標時刻の何分前からアラームを鳴らし始めるか。

例:

```text
60分前
```

### 6.5 アラーム間隔

起床ウィンドウ内で何分おきにアラームを鳴らすか。

例:

```text
5分間隔
```

## 7. 機能要件

### 7.1 起床プラン作成

ユーザーは、起きたい時刻を指定して起床プランを作成できる。

#### 必須入力

```text
- 起床目標時刻
- 日付または繰り返し条件
```

#### 初期値

```text
起床ウィンドウ: 60分
アラーム間隔: 5分
最終アラーム: 起床目標時刻に鳴らす
繰り返し: なし
```

#### 例

```text
起床目標時刻: 07:00
起床ウィンドウ: 60分
アラーム間隔: 5分

生成されるアラーム:
06:00
06:05
06:10
...
06:55
07:00
```

### 7.2 アラーム生成ルール

起床プラン作成時、アプリは以下のルールでアラーム発生予定を生成する。

```text
開始時刻 = 起床目標時刻 - 起床ウィンドウ
終了時刻 = 起床目標時刻
間隔 = アラーム間隔
```

生成対象には、原則として **起床目標時刻そのものを含める**。

#### 例

```text
起床目標: 08:00
ウィンドウ: 30分
間隔: 10分

生成:
07:30
07:40
07:50
08:00
```

### 7.3 起床ウィンドウ調整

ユーザーは、起床プランごとに起床ウィンドウを変更できる。

MVPでは自由入力よりもプリセット中心とする。

#### 推奨プリセット

```text
30分
45分
60分
90分
```

必要であれば詳細設定からカスタム入力を許可する。

### 7.4 アラーム間隔調整

ユーザーは、起床プランごとにアラーム間隔を変更できる。

#### 推奨プリセット

```text
3分
5分
10分
15分
```

#### 制約

極端に短い間隔は、OS制約、バッテリー消費、ストア審査、ユーザー体験上の問題があるため制限する。

MVPでは最小間隔を以下のどちらかにする。

```text
案A: 3分
案B: 5分
```

初期リリースでは **5分を標準** とする。

### 7.5 デフォルト設定

ユーザーは、新規作成時のデフォルト値を変更できる。

#### MVPで変更可能にする項目

```text
- 起床ウィンドウ
- アラーム間隔
- 曜日繰り返しの初期値
- バイブレーション有無
- アラーム音
```

#### 初期値

```text
起床ウィンドウ: 60分
アラーム間隔: 5分
曜日繰り返し: なし
バイブレーション: ON
アラーム音: デフォルト
```

### 7.6 一回限りの起床プラン

ユーザーは、特定の日付・時刻にだけ有効な起床プランを作成できる。

一回限りの起床プランは、最後のアラーム時刻を過ぎたあと、通常一覧から自動的に非表示にする。

#### 非表示タイミング

MVPでは以下を推奨する。

```text
最後のアラーム時刻 + 30分後
```

例:

```text
最終アラーム: 07:00
通常一覧から非表示: 07:30以降
```

内部的には、トラブル調査や履歴表示のため、短期間ログとして保持してもよい。

### 7.7 曜日繰り返し

ユーザーは、曜日単位で繰り返し起床プランを作成できる。

#### 対応する繰り返し

```text
- 毎日
- 平日
- 土日
- 任意の曜日指定
```

#### MVPで対応しない繰り返し

```text
- 隔週
- 毎月第n曜日
- 月末
- 祝日を除外
- カレンダー予定に応じた自動変更
```

### 7.8 次回だけスキップ

ユーザーは、繰り返し起床プランについて、次回分だけをスキップできる。

#### 例

```text
平日 07:00 の起床プランがある
明日だけ休みなので、次回だけスキップする
翌々日以降は通常通り鳴る
```

#### 仕様

```text
- スキップ対象は次に発火する1回分のWake Plan
- スキップ後、次回以降の繰り返し設定は維持される
- スキップ済み状態は一覧上で分かるように表示する
```

### 7.9 アラーム鳴動中の操作

アラーム鳴動中に表示する主操作は、現在鳴っているアラームを止める操作のみとする。

#### 表示する操作

```text
今のアラームを止める
```

#### 表示しない操作

```text
起きた
残りを全部止める
今日はもう鳴らさない
スヌーズ
```

理由は、ユーザーが一度目を覚ましても二度寝する可能性があるため。

#### 重要仕様

現在鳴っているアラームを止めても、同じ起床プラン内の未来のアラームはキャンセルされない。

```text
06:00のアラームを止める
→ 06:05以降は予定通り鳴る
```

### 7.10 起床プラン編集

ユーザーは、予定時刻前の起床プランを編集できる。

編集可能な項目:

```text
- 起床目標時刻
- 起床ウィンドウ
- アラーム間隔
- 繰り返し条件
- アラーム音
- バイブレーション
```

編集時は、既存のアラーム発生予定をすべてキャンセルし、新しい設定に基づいて再生成する。

#### 必須要件

```text
- 古いアラームが残らないこと
- 同じ時刻に重複アラームが生えないこと
- 編集後の発火予定をユーザーが確認できること
```

### 7.11 起床プラン削除

ユーザーは、起床プランを削除できる。

削除時は、その起床プランに紐づく未来のアラーム発生予定をすべてキャンセルする。

#### 確認ダイアログ

繰り返し設定がある場合は確認を表示する。

```text
この起床プランを削除しますか？
今後のアラームもすべて削除されます。
```

### 7.12 テストアラーム

ユーザーは、アプリが正しく鳴るか確認するためのテストアラームを実行できる。

#### 仕様

```text
- 30秒後または1分後にテストアラームを鳴らす
- 実際の鳴動画面、通知、音、バイブを確認できる
- テストアラームは起床プラン一覧には残さない
```

### 7.13 権限・設定チェック

アプリは、アラームが正常に鳴らない可能性がある状態を検知し、ユーザーに警告する。

#### MVPでチェックする項目

```text
- 通知権限
- 正確なアラーム実行に必要な権限
- サウンド設定
- バイブレーション設定
- OS側でアプリ通知が無効化されていないか
```

#### 将来的にチェックしたい項目

```text
- 省電力設定
- バッテリー最適化
- フルスクリーン表示可否
- 集中モード/おやすみモードとの関係
- ロック画面表示可否
```

#### プラットフォーム制約メモ

MVP前に、iOS/Androidそれぞれで以下を実機検証する。

```text
- アプリ終了中に指定時刻で鳴るか
- 端末ロック中に鳴るか
- 通知権限がない場合の挙動
- サイレントモード、集中モード、省電力設定での挙動
- 正確な時刻で複数アラームを連続実行できるか
- ストア審査上、アラーム用途として妥当か
```

## 8. 非機能要件

### 8.1 信頼性

アラームアプリにおいて最も重要なのは、指定時刻に鳴ることである。

#### 要件

```text
- アプリを閉じていても鳴ること
- 端末がロック中でも鳴ること
- 起床プラン編集後に古いアラームが残らないこと
- 権限不足時は明確に警告すること
- OS再起動後に必要なアラームを復元できること
```

### 8.2 分かりやすさ

ユーザーが設定内容を直感的に理解できること。

#### 表示例

```text
07:00に起きる
06:00から5分おきに鳴ります
合計13回
```

一覧では、個々のアラームをすべて並べるより、起床プラン単位で表示する。

```text
平日 07:00
06:00〜07:00 / 5分おき
```

### 8.3 誤操作防止

本アプリは二度寝防止が目的のため、未来のアラームを簡単に止められるUIを避ける。

#### 方針

```text
- アラーム鳴動中に「起きた」ボタンを置かない
- アラーム鳴動中に「残り全部停止」を置かない
- 削除やスキップは設定画面・詳細画面で行う
- 重要操作には確認を入れる
```

### 8.4 パフォーマンス

大量のアラーム発生予定が生成されても、アプリの動作が重くならないこと。

#### 例

```text
平日 07:00
60分前から5分間隔
1日13個
週5日で65個相当
```

内部的には、無制限に未来分を生成せず、必要な期間だけスケジュールする設計にする。

### 8.5 プライバシー

MVPではカレンダー連携を行わないため、カレンダー権限は要求しない。

#### 方針

```text
- 初回起動時に不要な権限を求めない
- MVPで必要な権限はアラーム/通知関連に限定する
- カレンダー連携は将来機能として、ユーザーが明示的に利用開始した時だけ権限を求める
```

## 10. 状態管理要件

### 10.1 Wake Plan状態

Wake Planは以下の状態を持つ。

```text
scheduled: 今後鳴る予定がある
active: 起床ウィンドウ中
finished: 一回限りで終了済み
skipped: 次回分がスキップされている
deleted: 削除済み
```

### 10.2 Alarm Occurrence状態

個々のアラーム発生予定は以下の状態を持つ。

```text
scheduled: 予定済み
ringing: 鳴動中
dismissed: ユーザーが今のアラームを止めた
missed: 何らかの理由で鳴動確認できなかった
expired: 過去分として期限切れ
cancelled: Wake Plan編集・削除によりキャンセル
```

### 10.3 重要な状態遷移

現在鳴っているアラームを止めても、Wake Planは終了しない。

```text
AlarmOccurrence: ringing → dismissed
WakePlan: active のまま
次のAlarmOccurrence: scheduled のまま
```

Wake Planが終了するのは、起床ウィンドウ内の最後のアラームが終了した後。

```text
最後のAlarmOccurrence終了
→ 一回限りWakePlanは finished
→ 繰り返しWakePlanは次回分を scheduled
```

## 12. 受け入れ条件

### 12.1 起床プラン作成

```text
Given ユーザーが07:00を起床目標に設定する
And 起床ウィンドウが60分
And アラーム間隔が5分
When 起床プランを保存する
Then 06:00〜07:00まで5分間隔のアラームが生成される
And 07:00のアラームも含まれる
```

### 12.2 現在のアラーム停止

```text
Given 06:00のアラームが鳴っている
When ユーザーが「今のアラームを止める」を押す
Then 06:00のアラームは停止する
And 06:05以降のアラームはキャンセルされない
```

### 12.3 起きたボタン非表示

```text
Given アラームが鳴っている
When アラーム鳴動画面を表示する
Then 「起きた」ボタンは表示されない
And 「残りを全部止める」ボタンも表示されない
```

### 12.4 起床プラン編集

```text
Given 07:00の起床プランが存在する
When ユーザーが起床目標を07:30に変更する
Then 古い06:00〜07:00のアラームはキャンセルされる
And 新しい06:30〜07:30のアラームが生成される
And 重複アラームは存在しない
```

### 12.5 一回限りプランの整理

```text
Given 一回限りの起床プランがある
And 最後のアラームが07:00に鳴る
When 07:30を過ぎる
Then そのWake Planは通常一覧から非表示になる
```

### 12.6 次回だけスキップ

```text
Given 平日07:00の繰り返しWake Planがある
When ユーザーが「次回だけスキップ」を押す
Then 次回分のアラームは鳴らない
And その次の対象日から通常通り鳴る
```

## 13. カレンダー連携の将来要件

MVPではカレンダー連携を含めない。  
ただし、将来的には以下の機能を検討する。

### 13.1 カレンダー予定表示

ユーザーのカレンダー予定を数日先まで表示し、予定から起床プランを作成できる。

例:

```text
明日 09:00 会議
→ 07:00に起きる起床プランを作成
```

### 13.2 予定開始のn時間前に起きる

予定ごと、またはデフォルトとして、予定開始の何時間前に起きるかを設定できる。

```text
出社予定の2時間前に起きる
旅行予定の3時間前に起きる
```

### 13.3 アラーム未設定通知

カレンダー予定があるにもかかわらず、関連する起床プランが存在しない場合に通知する。

ただし、誤通知が非常に起きやすいため、以下の条件が必要。

```text
- 対象カレンダーを選べる
- 終日予定を除外できる
- オンライン予定を除外できる
- 通知対象時間帯を限定できる
- ユーザーが明示的に有効化した場合のみ通知する
```

## 14. リスクと注意点

### 14.1 「鳴らない」リスク

アラームアプリとして最も致命的なリスクは、指定時刻に鳴らないことである。

#### 対策

```text
- 初回セットアップで権限チェック
- テストアラームを提供
- 権限不足時にホーム画面で警告
- OS設定変更後に再チェック
- 起床プラン保存時にスケジュール成功/失敗を確認
```

### 14.2 アラーム数が多すぎるリスク

60分前から5分間隔の場合、1回のWake Planで13個のアラームが生成される。

複数の繰り返しプランを持つと、管理すべき発火予定が増える。

#### 対策

```text
- 未来分を無制限に生成しない
- 直近数日分だけスケジュールする
- Wake Plan単位で管理する
- 編集時は必ず既存Occurrenceをキャンセルして再生成する
```

### 14.3 ユーザーが止めたくなるリスク

二度寝防止のため「起きたボタン」は置かないが、ユーザーが予定変更などで本当に止めたい場合もある。

MVPでは、アラーム鳴動画面には残り停止導線を置かない。  
ただし、起床プラン詳細画面から削除・無効化は可能にする。

#### 方針

```text
- 鳴動中の主導線には置かない
- 事前の編集、削除、スキップは可能
- 「起きた」という達成ボタンは作らない
```

### 14.4 カレンダー連携の複雑化リスク

カレンダー連携は便利だが、MVPに入れると初期体験が重くなる。

#### 理由

```text
- 権限要求が増える
- プライバシー説明が必要
- 予定の種類を判別する必要がある
- 誤通知が不快になりやすい
- プロダクトの焦点がぼやける
```

そのため、MVPでは明確にスコープ外とする。

## 17. MVPで最も重要な品質基準

MVPで守るべき品質基準は以下。

```text
1. 起きたい時刻を入れるだけで、迷わず起床プランが作れる
2. 生成されるアラーム時刻が分かりやすい
3. 1回止めても未来のアラームが止まらない
4. 起きたボタンが存在しない
5. 古いアラームや重複アラームが残らない
6. 権限不足で鳴らない状態を放置しない
7. アプリを閉じても、端末ロック中でも鳴ることを目指す
```

---

## 18. まとめ

MVPでは、カレンダー連携や高度な自動提案には進まず、以下の価値に集中する。

> 起きたい時刻を設定するだけで、前の時間帯に複数アラームが自動生成される。  
> 1回止めても次が鳴る。  
> 「起きたつもり」を信用しない。

この方針により、既存の汎用アラームやスヌーズ機能ではなく、**二度寝前提の段階起床アラーム**として差別化する。
